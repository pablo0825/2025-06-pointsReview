# 舊資料處理決策

本文件記錄舊 MongoDB 系統資料在第一版 PostgreSQL 重建專案中的處理決策。

## 決策

第一版 **不遷移舊 MongoDB 資料**。

PostgreSQL 新系統從乾淨資料庫開始，透過 migration 建立 schema，透過 seed 建立初始點數規則、測試資料與展示資料。

## 適用範圍

不遷移的舊資料包含：

- 舊申請資料。
- 舊帳號資料。
- 舊附件與簽名檔案。
- 舊點數資料。
- 舊寄信任務或通知紀錄。
- 舊審核紀錄。

新系統只保存上線後透過新流程建立的正式資料。

## 理由

不遷移舊資料的主要理由：

- 新資料模型新增申請版本、指導老師簽名、點數規則版本、學生班級快照、不可變點數流水帳與通用稽核紀錄；舊資料通常缺少這些完整語意。
- 硬把舊資料塞入新 schema，容易產生看似完整但實際不可追溯的歷史紀錄。
- 舊附件、簽名與審核紀錄若缺少一致的 storage key、版本關聯或簽核時間，會破壞新系統的資料完整性假設。
- 第一版目標是建立乾淨且可維護的新 PostgreSQL 設計，而不是兼容舊資料的不完整狀態。
- 展示與測試資料可由 seed 建立，不需要依賴舊 MongoDB。

## 舊資料保留方式

若仍需保留舊資料供備查，建議在新系統之外處理：

- 保留舊 MongoDB dump。
- 匯出舊系統報表為 CSV、PDF 或唯讀檔案。
- 將舊系統以唯讀模式短期保留，直到行政確認不再需要。

舊資料備查不進入新 PostgreSQL schema，也不透過新系統 API 查詢。

## 新系統初始化

新系統初始化方式：

1. 建立乾淨 PostgreSQL database。
2. 執行完整 migration。
3. 執行必要 seed：
   - 初始點數規則。
   - 測試或開發環境帳號。
4. 使用維運指令建立第一位管理員。
5. 上線後只接受新系統建立的新申請與新點數資料。

Production 不自動匯入舊申請、舊點數或舊帳號。

## 若未來重新考慮遷移

若未來因法規、行政或業務需求必須遷移舊資料，應另開獨立資料遷移專案，不應混入第一版 schema migration。

未來遷移專案至少需要重新設計：

- 舊 MongoDB collection 到新 PostgreSQL table 的 mapping。
- 舊狀態到新狀態的轉換。
- 舊申請缺少版本資料時的補值規則。
- 舊簽名與附件檔案的 storage key 產生方式。
- 舊點數轉成 `student_point_transactions` 的快照規則。
- 舊資料缺少點數規則版本時的歸屬規則。
- dry run、資料驗證與遷移報告。
- 遷移期間停機或唯讀策略。

在上述專案完成前，不應把舊資料寫入新 PostgreSQL 正式資料表。
